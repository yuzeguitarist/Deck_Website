1f:T521a,<hr>
<p>剪贴板管理器能看到用户复制的所有内容——密码、银行卡号、身份证号、机密文件。<a href="https://deckclip.app">Deck</a> 是一款基于 Swift / SwiftUI 构建的 macOS 原生剪贴板管理应用，这种可见性意味着一项义务：应用必须保护它接触到的每一份数据。</p>
<p>Deck 的做法是纵深防御——多层相互重叠的保护机制，覆盖过滤、加密、访问控制和隐蔽通信。</p>
<h3 id="威胁模型">威胁模型</h3>
<p>Deck 的安全设计针对三类场景：（1）<strong>数据泄漏</strong>——密码或卡号等敏感内容被意外保存到剪贴板历史；（2）<strong>本地未授权访问</strong>——有人在未锁定的 Mac 上直接查看剪贴板数据库；（3）<strong>传输截获</strong>——剪贴板数据在局域网共享或粘贴到不可信渠道时被拦截。架构中的每一层都对应其中一个或多个威胁场景。</p>
<hr>
<h2 id="1-多层敏感内容过滤管线">1. 多层敏感内容过滤管线</h2>
<p>当用户从聊天窗口复制了一个银行卡号，过滤管线会在几毫秒内拦截——在这串数字触及数据库之前。</p>
<p>管线位于 <code>ClipboardService.swift</code>，在 <code>checkForChangesInternal()</code> 检测到剪贴板变化时触发。四道检查按从快到慢的顺序依次执行，最廉价的检查充当快速出口。</p>

<hr>
<h2 id="2-aes-gcm-全链路加密">2. AES-GCM 全链路加密</h2>
<p>过滤管线处理的是已知的敏感模式，但普通剪贴板内容——笔记、代码片段、私人消息——同样需要静态加密保护。Deck 在整个数据链路中使用 AES-256-GCM，与 TLS 1.3 相同的认证加密标准。</p>
<h3 id="密钥管理">密钥管理</h3>
<p>256 位对称密钥在首次使用时生成，并通过 <code>getOrCreateEncryptionKey()</code> 存入 macOS Keychain。密钥永远不会触及文件系统。为减少 Keychain 访问开销，密钥在内存中缓存 60 秒。</p>
<h3 id="数据库列级加密">数据库列级加密</h3>
<p>当安全模式（<code>securityModeEnabled</code>）开启时，以下数据库列在写入 SQLite 之前会被逐列加密：</p>
<p>原始剪贴板数据
搜索文本
来源应用名
自定义标题
语义向量</p>
<div class="code-block-wrapper"><button class="copy-btn" aria-label="Copy code"><svg width="14" height="14" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round"><rect width="14" height="14" x="8" y="8" rx="2" ry="2"/><path d="M4 16c-1.1 0-2-.9-2-2V4c0-1.1.9-2 2-2h10c1.1 0 2 .9 2 2"/></svg></button><div class="code-lang-label">swift</div><pre><code class="hljs language-swift"><span class="hljs-comment">// Encryption / 加密</span>

<span class="hljs-keyword">return</span> sealedBox.combined  <span class="hljs-comment">// nonce + ciphertext + tag</span>

<span class="hljs-comment">// Decryption / 解密</span>
<span class="hljs-keyword">let</span> sealedBox <span class="hljs-operator">=</span> <span class="hljs-keyword">try</span> <span class="hljs-type">AES</span>.<span class="hljs-type">GCM</span>.<span class="hljs-type">SealedBox</span>(combined: encryptedData)

<h3 id="blob-文件加密">Blob 文件加密</h3>
<p>超过 512 KB（<code>largeBlobThreshold</code>）的剪贴板条目以独立文件而非数据库内联方式存储。在安全模式下，这些 Blob 文件在写入磁盘前同样经过 AES-GCM 加密。</p>
<h3 id="局域网共享加密">局域网共享加密</h3>
<p>当 Deck 通过 MultipeerConnectivity 在设备间同步剪贴板内容时，共享数据使用 TOTP 验证后协商的会话密钥加密，与加密链路其他部分使用相同的 CryptoKit <code>AES.GCM</code> 原语。</p>
<h3 id="性能代价">性能代价</h3>
<p>Apple Silicon 上的 AES-GCM 有硬件加速，对于典型剪贴板条目（文本、小图片）来说单条加解密的开销可以忽略。真正的代价体现在别处：加密列会破坏 SQLite FTS5 和向量索引，迫使搜索退化为先解密再匹配的内存扫描（详见第 6 节）。60 秒密钥内存缓存和 300 条 <code>searchTextCache</code> 就是为了让这条退化路径保持可用的响应速度。</p>
<hr>
<h2 id="3-touch-id-生物识别认证">3. Touch ID 生物识别认证</h2>
<p>加密覆盖了静态数据和传输中的数据，但如果有人直接走到一台未锁定的 Mac 前打开 Deck，加密就帮不上忙了。生物识别认证通过物理身份验证填补了这个缺口。</p>
<p>Deck 使用 <code>LocalAuthentication</code> 框架，采用两级策略：</p>
<p>回退**：<code>.deviceOwnerAuthentication</code> — system password when biometrics are unavailable / 生物识别不可用时使用系统密码</p>
<p>提供两种超时模式：</p>
<p>自动锁定**：5 minutes of inactivity triggers re-authentication / 5 分钟无操作后要求重新认证
每次认证**：authentication required on every app launch (<code>authEveryTime</code>) / 每次打开应用都要求认证</p>
<hr>
<h2 id="4-零宽字符隐写">4. 零宽字符隐写</h2>
<p>以上各层保护的是 Deck 内部的数据。隐写术则将保护向外延伸——用户可以在分享前将信息隐藏在普通文本中，这样即使传输渠道被窃听，内容也不会被识别。</p>
<p>零宽字符是不占据可见空间的 Unicode 码位。Deck 使用其中四个字符进行 2 bit/字符的编码：</p>

<p>编码流程：</p>

<hr>
<h2 id="5-图片-lsb-隐写">5. 图片 LSB 隐写</h2>
<p>当需要更大的载荷容量，或文本不适合作为载体时，Deck 支持图片最低有效位（LSB）隐写。</p>
<p>该算法修改每个非透明像素各 RGB 通道的最低位，每像素可存储 3 bit。一张 1920×1080 无透明像素的图片可承载约 760 KB 的隐藏数据——对大多数消息来说绰绰有余。</p>
<p>其流程与零宽字符方案类似：</p>
<p>明文 → AES-GCM 加密
添加 <code>DECKSTG1</code> 魔数头部（14 字节，含载荷长度）
将加密后的 bit 写入像素 RGB 通道最低位
通过 <code>CGContext</code> 重新编码图像</p>
<p>透明像素（alpha &lt; 1）被跳过——修改它们会产生可见的视觉伪影。载体图片上限约为 25 MB。</p>
<h3 id="隐写密钥管理">隐写密钥管理</h3>
<p>隐写加密使用专用密钥，通过 HKDF-SHA256 从主密钥派生（<code>SteganographyKeyStore.swift</code>）。这种分离确保一个密钥被泄露时不会波及另一个。</p>
<hr>

<p>速度不如 FTS5，但在不向磁盘写入明文搜索索引的前提下保留了功能可用性。</p>
<hr>
<h2 id="7-各层如何协同工作">7. 各层如何协同工作</h2>
<p>每一层对应不同的威胁面。过滤将敏感数据挡在系统之外；加密保护已存储的内容；生物识别阻止未经授权的物理访问；隐写术保护离开 Deck 之后的数据。单独任何一层都不够用——它们的价值在于重叠。</p>

<p>Search makes or breaks a clipboard manager. You might copy hundreds of things a day — if you can&#39;t quickly find that one snippet from this morning, the history is just noise.</p>
<p>Cloud-based apps can offload this to Elasticsearch or Algolia. <a href="https://deckclip.app">Deck</a> doesn&#39;t have that option — it&#39;s a macOS-native clipboard manager built with Swift and SwiftUI, and all data stays on the user&#39;s machine. No server, no network calls, no telemetry. The search system has to be built entirely on top of SQLite and Apple&#39;s native frameworks.</p>
<p>This post covers how that works.</p>
<hr>
<h2 id="the-challenge">The Challenge</h2>
<p>Local search is deceptively hard. Consider what a clipboard manager must handle:</p>
<ul>
<li><strong>CJK text</strong> that has no whitespace delimiters</li>
<li><strong>Typos and fuzzy recall</strong> — the user types &quot;clpboard&quot; and still expects to find &quot;clipboard&quot;.</li>
<li><strong>Semantic intent</strong> — searching &quot;network error&quot; should surface entries about &quot;connection timeout&quot;.</li>
<li><strong>Sub-second latency</strong> — search must feel instant, even with tens of thousands of entries.</li>
<li><strong>Zero resource waste</strong> — a background app cannot burn CPU or memory.</li>
</ul>
<p>Deck&#39;s answer is a <strong>five-layer search pipeline</strong> — each layer catches what the previous one missed.</p>

<h2 id="layer-1-fts5-full-text-search-with-trigram-cjk-support">Layer 1: FTS5 Full-Text Search with Trigram CJK Support</h2>
<p>SQLite&#39;s FTS5 is the workhorse of Deck&#39;s search. But out of the box, FTS5&#39;s default tokenizer splits on whitespace</p>
<p>The fix: <strong>trigram tokenization</strong>. Instead of splitting on word boundaries, trigram slides a 3-character window across the text, generating overlapping tokens. <code>剪贴板管理</code> becomes <code>剪贴板</code>, <code>贴板管</code>, <code>板管理</code> — and substring search Just Works.</p>

<h2 id="layer-2-bitap-style-fuzzy-matching-fusesearch">Layer 2: Bitap-Style Fuzzy Matching (FuseSearch)</h2>
<p>FTS5 is fast but literal — it won&#39;t help when the user mistypes &quot;screnshot&quot; for &quot;screenshot&quot;. That&#39;s where <code>FuseSearch</code> comes in.</p>
<p>The algorithm works in two phases:</p>
<ol>
<li><strong>Exact scan</strong> — <code>O(n)</code> substring search. If it hits, return immediately with a perfect score.</li>
<li><strong>Fuzzy match</strong> — Walk through the text character by character, greedily matching each pattern character in order. Track the positions and gaps.</li>
</ol>

<h2 id="layer-3-semantic-vector-search-with-nlembedding">Layer 3: Semantic Vector Search with NLEmbedding</h2>

<h2 id="layer-4-sqlite-vec-persistent-vector-index">Layer 4: sqlite-vec — Persistent Vector Index</h2>

<h2 id="layer-5-slash-rule-filters-unicode-private-use-area-encoding">Layer 5: Slash-Rule Filters — Unicode Private Use Area Encoding</h2>

<h2 id="what-we-learned">What We Learned</h2>
<p>No single search technique covers every case. FTS5 handles most queries well, but it falls apart on typos and can&#39;t do semantic matching. Fuzzy search catches typos but gets noisy on long texts. Semantic search understands intent but costs memory and CPU. Stacking them together and letting each one cover the others&#39; gaps turned out to be more practical than trying to build one perfect search algorithm.</p>
<p>CJK support was a decision we made early on. FTS5&#39;s default tokenizer splits on whitespace, which means Chinese, Japanese, and Korean text is effectively unsearchable. Trigram tokenization fixes this at the cost of ~3× index size — a trade-off we&#39;re comfortable with.</p>
<p>The resource management side took more iteration than expected. A clipboard manager runs all day, and embedding models are not small. We went through a few rounds of tuning before landing on the current pattern: lazy-load the model, cache up to 400 vectors, release the model after 30 seconds of idle, and block matrix multiplication into 256-row chunks to keep memory flat.</p>
<p>Everything described in this post runs locally on the user&#39;s Mac. No data leaves the device, no cloud API is called. That was a constraint from the start, not a feature we added later.</p>

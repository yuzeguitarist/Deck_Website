<p>A clipboard manager sees everything the user copies — passwords, bank card numbers, ID numbers, confidential documents. <a href="https://deckclip.app">Deck</a> is a macOS-native clipboard manager built with Swift and SwiftUI, and this visibility comes with an obligation: the app must protect every piece of data it touches.</p>
<p>Deck&#39;s approach is defense in depth — multiple overlapping layers of protection covering filtering, encryption, access control, and covert communication.</p>
<h3 id="threat-model">Threat Model</h3>
<p>Deck&#39;s security design targets three scenarios: (1) <strong>data leakage</strong> — sensitive content like passwords or card numbers accidentally saved to clipboard history; (2) <strong>local unauthorized access</strong> — someone with physical access to an unlocked Mac browsing the clipboard database; (3) <strong>in-transit interception</strong> — clipboard data exposed during LAN sharing or when pasted into untrusted channels. Each layer in the architecture maps to one or more of these threats.</p>
<hr>
<h2 id="1-the-sensitive-content-filtering-pipeline">1. The Sensitive Content Filtering Pipeline</h2>
<p>When a user copies a bank card number from a chat window, the filtering pipeline intercepts it within milliseconds — before the number touches the database.</p>
<p>The pipeline lives in <code>ClipboardService.swift</code>, triggered inside <code>checkForChangesInternal()</code> whenever a clipboard change is detected. It runs four checks in sequence, ordered from fastest to slowest so the cheapest checks act as early exits.</p>

<p>The fastest check — zero parsing required. Password managers like 1Password and the macOS Keychain tag their clipboard entries with special pasteboard types:</p>
<ul>
<li><code>org.nspasteboard.ConcealedType</code> (community standard</li>
<li><code>com.apple.password</code></li>
<li><code>com.apple.securetext</code></li>

<p>If any of these types appear in <code>pasteboard.types</code>, Deck immediately skips the entry. No content is read, no content is stored.</p>

<h2 id="2-aes-gcm-full-chain-encryption">2. AES-GCM Full-Chain Encryption</h2>
<p>The filtering pipeline handles known sensitive patterns, but ordinary clipboard content — notes, code snippets, personal messages — also needs protection at rest. Deck encrypts the entire data chain with AES-256-GCM, the same authenticated encryption standard used in TLS 1.3.</p>
<h3 id="key-management">Key Management</h3>
<p>A 256-bit symmetric key is generated on first use and stored in the macOS Keychain via <code>getOrCreateEncryptionKey()</code>. The key never touches the filesystem. To reduce Keychain access overhead, the key is cached in memory for 60 seconds.</p>
<h3 id="column-level-database-encryption">Column-Level Database Encryption</h3>
<p>When Security Mode is enabled (<code>securityModeEnabled</code>), the following database columns are encrypted individually before writing to SQLite:</p>
<ul>
<li><code>data</code> — raw clipboard bytes</li>
<li><code>search_text</code> — searchable text</li>
<li><code>app_name</code> — source application</li>
<li><code>custom_title</code> — user-assigned title</li>
<li><code>embedding</code> — semantic vector</li>
</ul>

<h3 id="blob-file-encryption">Blob File Encryption</h3>
<p>Clipboard items larger than 512 KB (<code>largeBlobThreshold</code>) are stored as standalone files rather than inline in the database. In Security Mode, these blob files are also AES-GCM encrypted before being written to disk.</p>
<h3 id="lan-sharing-encryption">LAN Sharing Encryption</h3>
<p>When Deck syncs clipboard content between devices via MultipeerConnectivity, the shared data is encrypted with a session key negotiated after TOTP verification, using the same CryptoKit <code>AES.GCM</code> primitives as the rest of the encryption chain.</p>
<h3 id="performance-cost">Performance Cost</h3>
<p>AES-GCM on Apple Silicon is hardware-accelerated, so the per-item encryption overhead is negligible for typical clipboard entries (text, small images). The main cost shows up elsewhere: encrypted columns break SQLite FTS5 and vector indexes, forcing search to fall back to in-memory decryption-then-match (covered in Section 6). The 60-second in-memory key cache and the 300-entry <code>searchTextCache</code> exist specifically to keep this fallback path responsive.</p>
<hr>
<h2 id="3-touch-id-biometric-authentication">3. Touch ID Biometric Authentication</h2>
<p>Encryption covers data at rest and in transit, but it doesn&#39;t help if someone walks up to an unlocked Mac and opens Deck directly. Biometric authentication closes that gap with a physical identity check.</p>
<p>Deck uses the <code>LocalAuthentication</code> framework with a two-tier policy:</p>
<ol>
<li><strong>Primary / 首选</strong>：<code>.deviceOwnerAuthenticationWithBiometrics</code> — Touch ID, Face ID, or Optic ID</li>
<li>**Fallback</li>
</ol>
<p>Two timeout modes are available:</p>
<ul>
<li>**Auto-lock</li>
<li>**Every-time</li>
</ul>

<h2 id="5-image-lsb-steganography">5. Image LSB Steganography</h2>
<p>For richer payloads or when text isn&#39;t the right carrier, Deck supports Least Significant Bit (LSB) steganography in images.</p>
<p>The algorithm modifies the lowest bit of each RGB channel in every non-transparent pixel, yielding 3 bits of storage per pixel. A 1920×1080 image with no transparency can carry roughly 760 KB of hidden data — more than enough for most messages.</p>
<p>The pipeline mirrors the zero-width approach:</p>
<ol>
<li>Plaintext → AES-GCM encryption</li>
<li>Prepend <code>DECKSTG1</code> magic header (14 bytes, including payload length)</li>
<li>Write encrypted bits into the LSBs of pixel RGB channels</li>
<li>Re-encode the image via <code>CGContext</code></li>
</ol>
<p>Transparent pixels (alpha &lt; 1) are skipped — modifying them would create visible artifacts. The carrier image limit is approximately 25 MB.</p>
<h3 id="steganography-key-management">Steganography Key Management</h3>
<p>Steganographic encryption uses a dedicated key, derived from the master key via HKDF-SHA256 (<code>SteganographyKeyStore.swift</code>). This separation ensures that compromising one key doesn&#39;t compromise the other.</p>

<h2 id="7-how-the-layers-fit-together">7. How the Layers Fit Together</h2>
<p>Each layer addresses a different threat surface. Filtering keeps sensitive data out of the system; encryption protects what does get stored; biometrics prevents unauthorized physical access; steganography protects data after it leaves Deck. None of them is sufficient on its own — the value is in the overlap.</p>

<li><strong>Filtering</strong> intercepts passwords, card numbers, and ID numbers at the entry point, keeping them out of storage entirely.</li>
<li><strong>Encryption</strong> covers database columns, on-disk blob files, and LAN transfers — protecting all stored and in-transit data.</li>
<li><strong>Biometrics</strong> requires Touch ID or a password before anyone can view clipboard history on an unlocked Mac.</li>
<li><strong>Steganography</strong> adds an invisible encryption layer for users sharing content through untrusted channels.</li>
<li>When Security Mode is on, <strong>search automatically degrades</strong> to an in-memory scan, avoiding plaintext indexes on disk.</li>
</ul>
<p>Deck reads everything the user copies. The architecture above makes sure that visibility doesn&#39;t become a liability — each layer reduces a different category of risk, and together they cover the gaps that any single mechanism would leave open.</p>

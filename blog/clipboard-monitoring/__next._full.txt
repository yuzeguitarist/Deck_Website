1e:T79fe,<p><strong>From 0.5s to 15s — How Deck Makes Clipboard Monitoring Both Responsive and Power-Efficient</strong></p>
<hr>
<p>The core of clipboard monitoring on macOS is straightforward: poll <code>NSPasteboard.general.changeCount</code> on a timer, check if the value changed, read the new content. A weekend prototype can get this working.</p>
<p>But a clipboard manager that runs 10+ hours a day needs to handle things the prototype ignores — battery impact from constant polling, 200 MB screenshots that blow up memory, Figma&#39;s non-standard clipboard format, and IDE source tracking across five editor families. This post covers how Deck handles these problems: adaptive polling, content-type parsing, OOM protection, IDE integration, and Figma-aware clipboard detection.</p>
<hr>
<h2 id="1-adaptive-polling-doing-less-when-it-matters">1. Adaptive Polling: Doing Less When It Matters</h2>
<p>The naive approach is a fixed-interval timer — say, every 0.5 seconds. That&#39;s responsive, but it&#39;s also 7,200 wakeups per hour. On a laptop running on battery, that&#39;s unacceptable.</p>
<p>Deck solves this with an <strong>adaptive polling mechanism</strong> that dynamically adjusts the interval based on three signals: user activity, power state, and thermal pressure.</p>

<h3 id="the-three-level-text-fallback-chain">The Three-Level Text Fallback Chain</h3>
<p>When the primary type can&#39;t be parsed, Deck doesn&#39;t give up. It walks a three-level fallback chain:</p>
<ol>
<li><p><strong><code>resolveHTMLTextPayload()</code></strong> — Checks for HTML content or Figma markup first.</p>
</li>
<li><p><strong><code>resolveUTF16PlainText()</code></strong> — Tries <code>public.utf16-external-plain-text</code>, then <code>public.utf16-plain-text</code>.</p>
</li>
<li><p><strong><code>resolveGenericTextPayload()</code></strong> — Iterates all types conforming to <code>UTType.text</code>.</p>
</li>
<li><p><strong><code>resolveHTMLTextPayload()</code></strong></p>
</li>
<li><p><strong><code>resolveGenericTextPayload()</code></strong></p>
</li>
</ol>
<p>An exclusion list of ~20 types (including <code>.string</code>, <code>.rtf</code>, <code>.html</code>, <code>.pdf</code>, various WebKit/Chromium internal types, and macOS system types) prevents the fallback chain from re-processing types that were already tried or are known to be noise.</p>
<p>A practical optimization: when an RTFD item also carries a plain text representation, Deck prefers the plain text — avoiding the overhead of RTFD decoding when the content is textual anyway.</p>

<h2 id="6-figma-clipboard-detection">6. Figma Clipboard Detection</h2>
<p>Figma runs as a Chromium-based desktop app, and its clipboard behavior reflects that. When you copy a frame or component in Figma, the pasteboard receives what looks like standard HTML — but embedded inside it is base64-encoded design data wrapped in custom <code>(figma)</code> / <code>(figmeta)</code> markers. Most clipboard managers treat this as HTML or plain text, stripping the Figma-specific data entirely. When you paste it back, the design structure is gone — you get a flat image or garbled text instead of the original editable component.</p>
<p>Deck detects Figma content via <code>containsFigmaMarker(in:)</code>, which scans the HTML for these markers. Once detected, a multi-step extraction pipeline kicks in:</p>
<ol>
<li><p><strong><code>resolveFigmaPayload()</code></strong> — Entry point; only runs for unresolved types.</p>
</li>
<li><p><strong><code>extractFigmaPayload()</code></strong> — Searches <code>UnsupportedPasteboardPayload.items</code> for Figma HTML.</p>
</li>
<li><p><strong><code>extractFigmaPayloadFromPlistFallback()</code></strong> — Falls back to plist parsing if needed.</p>
</li>
<li><p><strong><code>buildFigmaPayload()</code></strong> — Extracts <code>(figma)</code> and <code>(figmeta)</code> base64 data from the HTML.</p>
</li>
<li><p><strong><code>resolveFigmaPayload()</code></strong></p>
</li>
<li><p><strong><code>extractFigmaPayloadFromPlistFallback()</code></strong></p>
</li>
<li><p><strong><code>buildFigmaPayload()</code></strong></p>
</li>
</ol>

<h2 id="wrapping-up">Wrapping Up</h2>
<p>Most of these systems interact with each other. Thermal pressure changes the poll interval, which changes how quickly ⌘C interception matters. A Figma paste can carry 20+ pasteboard types, hitting both the content priority ladder and the OOM guards simultaneously. IDE anchor capture has to complete within the 80ms window before the next poll cycle overwrites the Accessibility state.</p>
<p>There are areas we&#39;re still improving. The Accessibility API can be slow on certain apps (Electron-based editors with deep DOM trees), and we&#39;re experimenting with caching strategies to reduce AX query frequency. We&#39;re also looking into using <code>NSPasteboard</code> observation APIs if Apple exposes more event-driven hooks in future macOS versions, which could reduce polling overhead further.</p>
